Issue Summary
- On desktop, project card alignment appeared correct on initial load.
- After navigating to a project link and returning with the browser Back button, card alignment could become incorrect until a full refresh.
- Sentinel-driven header state (theme and active nav underline) could fail to update after history restore.

Root Cause
- Project card layout used JavaScript breakpoint hooks to render different component trees for mobile vs desktop.
- Browser back-forward cache (BFCache) can restore page state without a full remount lifecycle, which can leave JS-derived layout state stale.
- Header sentinel behavior depended primarily on observer/mount timing and was not explicitly resynced for restore scenarios.
- Header setup could become brittle when sentinel elements were temporarily unavailable at setup time; lifecycle-level early bailout patterns prevent later observer/state recovery.

Fix Applied
1. Project card architecture
- File: src/components/project/projects/reuseables/projectCard/ProjectCard.tsx
- Switched to one stable markup structure for all viewport sizes.
- Removed JS media-query branching from card rendering.

2. Responsive behavior in CSS
- File: src/components/project/projects/reuseables/projectCard/styles/projectCard.module.css
- Moved viewport responsiveness and odd/even layout direction into CSS media queries and classes.
- Kept layout deterministic across restores by avoiding JS-driven structural swaps.

3. Sentinel/nav resync hardening
- File: src/components/project/header/Header.tsx
- Added explicit nav state recalculation on:
  - scroll
  - resize
  - pageshow (history restore/BFCache return path)
- Observer is rebound and nav state is recalculated after restore.
- Empty-sentinel guards are kept inside bind/resolve runtime paths (not as one-time effect lifecycle bailout), so the header can recover when sentinels are available again.

Why this works
- CSS-driven layout remains stable across hydration and BFCache restores.
- Header state no longer relies only on initial observer callbacks; it can recompute from current scroll/sentinel positions when the page is restored.
- Sentinel guard placement now avoids killing the effect lifecycle while still safely no-oping when sentinel lists are temporarily empty.

Validation
- TypeScript check passed: npm run typecheck
- Behavior to verify manually:
  1) Open a project link, then Back: card alignment should remain correct.
  2) Scroll across sections: nav theme and active underline should update consistently.
  3) If sentinels are temporarily unavailable during initial setup, confirm header theme/underline recover once scrolling/resizing/pageshow triggers rebind/resync.
